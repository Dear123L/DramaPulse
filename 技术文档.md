## 1. 项目概述

DramaPulse 是一款 Android 短剧互动 App，核心创新是在短剧播放过程中通过 AI 识别情绪高光点，在关键时刻触发分支选项，让观众决定剧情走向，实现"沉浸式互动观看"。

本文档覆盖后端服务的技术选型、模块拆解、核心流程及数据库设计，作为项目交付与后续维护的技术依据。

## 2. 核心模块技术选型

### 2.1 技术栈总览

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| Web 框架 | FastAPI 0.110+ | 原生异步支持，自动生成 OpenAPI 文档，性能优于 Flask/Django |
| ASGI 服务器 | Uvicorn | FastAPI 官方推荐，轻量高性能 |
| 数据库 | MySQL 8.0 | 关系型数据，支持事务，适合结构化剧情数据 |
| 数据库驱动 | PyMySQL | 纯 Python 实现，兼容性好 |
| AI 接口 | OpenAI SDK（兼容模式） | 接入字节跳动火山方舟（豆包 AI），OpenAI 兼容接口，零适配成本 |
| 数据校验 | Pydantic v2 | FastAPI 内置，类型安全，自动生成 API 文档 |
| 配置管理 | python-dotenv | 环境变量分离，Key 不进代码仓库 |
| 图片处理 | Pillow (PIL) | 无 AI Key 时的占位图生成，轻量可靠 |
| 静态文件 | FastAPI StaticFiles | 挂载多媒体资源，无需 Nginx 前置 |

### 2.2 为什么不用 Java / Flask / Django

- **不用 Java**：Python 在 AI 接口调用、脚本化数据处理方面开发效率更高，团队 AI 全栈比赛时间有限，快速迭代优先
- **不用 Flask**：Flask 同步阻塞，无原生异步支持，不适合 AI 调用的高延迟场景；且需手动集成 Swagger，开发效率低
- **不用 Django**：Django 过重，内置 ORM / Admin 等大量不急需的功能，引入不必要的复杂度

## 3. 模块拆解

### 3.1 目录结构

```
backend/
├── main.py                        # 服务入口，路由注册，CORS/静态文件挂载
├── config.py                      # 配置管理中心（从 .env 加载，含 SERVER_BASE_URL）
├── database.py                    # 数据库连接池（PyMySQL）
├── models.py                      # 数据模型 & API 响应格式化（含 interaction_type/ai_danmaku）
├── ai_service.py                  # AI 调用封装（豆包 AI 兼容接口）
├── api/                           # 接口路由模块
│   ├── __init__.py
│   ├── episodes.py                # 剧集/高光点查询路由
│   ├── ai_branch.py               # 方案A：纯文字分支续写
│   ├── ai_branch_image.py         # 方案B：文字 + AI 配图
│   ├── ai_branch_video_prepared.py # 方案C：预渲染视频分支
│   ├── ai_branch_options.py       # AI 分支选项生成
│   └── ai_interaction.py          # AI 互动类型分析
├── scripts/                       # 工具脚本（非运行依赖）
│   ├── mock_analyze_interaction.py # 模拟生成互动类型数据
│   └── assign_images.py           # 预渲染图片写入数据库
├── doc/                           # 对接文档
│   └── AiInteraction.openapi.json # AI互动分析接口 OpenAPI 规范
├── multimodal_assets/
│   └── ep67/                      # 第67集多媒体素材（图片）
├── frontend/                      # 前端验证页面（App 开发期间联调用）
├── requirements.txt               # Python 依赖清单
├── .env.example                   # 环境变量模板
├── .gitignore                     # Git 忽略规则
└── README.md                      # 项目使用说明
```

### 3.2 模块职责说明

#### `main.py` — 服务入口
- 初始化 FastAPI 应用，配置 CORS（允许 Android App 跨域调用）
- 挂载两个静态目录：`/static`（前端资源）、`/assets`（AI 生成多媒体）
- 注册全部 6 个路由模块（episodes、ai_branch、ai_branch_image、ai_branch_video_prepared、ai_branch_options、ai_interaction）
- 提供健康检查（`GET /`）和 API 摘要（`GET /api/summary`）

#### `config.py` — 配置管理
- 通过 `python-dotenv` 从 `.env` 文件加载敏感配置
- 集中管理：数据库地址、AI API Key、AI 接口 Base URL、服务端口、SERVER_BASE_URL
- **关键点**：所有敏感值不设默认值，必须从 `.env` 注入，防止误提交
- **SERVER_BASE_URL**：后端返回给前端的完整URL前缀，确保 image_url 等字段包含完整可访问地址

#### `database.py` — 数据层
- 封装 `PyMySQL` 连接，提供 `get_db()` 方法
- 由 `init_db()` 在启动时自动建表
- 所有业务模块通过 `database.py` 获取连接，避免连接泄漏

#### `ai_service.py` — AI 服务封装
- 统一封装豆包 AI（火山方舟）调用逻辑
- 使用 OpenAI SDK 兼容模式：`base_url` 指向火山方舟 endpoint
- 提供 `generate_branch_text()` 和 `generate_image()` 两个核心方法
- 内置缓存：相同高光点 + 分支组合只调用一次 AI，结果写入 `branch_results` 表

#### `api/episodes.py` — 剧集 & 高光点查询
- 接口：`GET /episodes`、`GET /episodes/{episode_no}`、`GET /episodes/{episode_no}/highlights`、`GET /episodes/{episode_no}/highlights/{id}`
- **App 核心接口**：高光点列表返回 `interaction_type`（1=手电/2=心跳/3=AI弹幕）和 `ai_danmaku`（弹幕JSON列表）

#### `api/ai_branch.py` — 方案A（纯文字）
- 接口：`POST /ai/branch`
- 流程：接收 `highlight_id` + `branch_id` → 查缓存 → 无缓存则调用 AI → 写入缓存 → 返回
- 前端降级链路的第一备选（最快、最稳定）

#### `api/ai_branch_image.py` — 方案B（文字 + AI 配图）
- 接口：`POST /ai/branch-with-image`
- 流程：同方案A → 额外调用文生图 API → 保存图片到 `multimodal_assets/ep67/` → 返回图片 URL
- 图片命名规范：`hl{highlight_id}_branch{A/B/C}.png`
- **降级策略**：无文生图 Key 时，从预设图片池随机返回一张预设图
- `image_url` 由 `SERVER_BASE_URL` 拼接为完整 URL 返回给前端

#### `api/ai_branch_video_prepared.py` — 方案C（预渲染视频）
- 接口：`POST /ai/branch-video`
- 流程：不实时生成视频，直接返回预先渲染好的视频文件路径
- 视频文件存放在 `frontend/branch_media/ep67/`，通过 `/static` 路由访问
- **实际使用方案**：答辩演示用，用户体验最佳，无实时生成延迟

#### `api/ai_branch_options.py` — AI 分支选项生成
- 接口：`POST /ai/branch-options/{highlight_id}`
- 调用豆包 AI 根据场景描述生成分支选项

#### `api/ai_interaction.py` — AI 互动类型分析
- 接口：`POST /ai/analyze-interaction/{highlight_id}`（单个）、`POST /ai/analyze-interaction/batch/{episode_no}`（批量）
- 调用豆包 AI 分析高光点最适合的互动类型（手电/心跳/AI弹幕）
- 如果互动类型为 AI 弹幕(3)，同时生成弹幕内容
- 批量分析加 `time.sleep(0.5)` 防限流
- AI 调用失败时降级返回默认类型（心跳）

#### `models.py` — 数据模型 & API 响应格式化
- 提供 `highlight_to_api()` 函数格式化高光点返回数据
- 含 `_parse_danmaku()` 辅助函数安全解析 `ai_danmaku` JSON 字段
- 返回字段包含 `interaction_type` 和 `ai_danmaku`

## 5. 数据库设计

### 5.1 表关系概览

```
episodes（剧集）
    │ 1:N
    ▼
highlights（高光点）
    │ 1:N
    ▼
branch_results（分支续写缓存）
```

### 5.2 各表字段说明

#### `episodes` — 剧集表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AI | 主键 |
| episode_no | INT UNIQUE | 集数编号（如 67） |
| title | VARCHAR | 剧集标题 |
| theme | VARCHAR | 主题标签 |
| total_duration | INT | 视频总时长（秒） |
| status | ENUM | 分析状态：pending/analyzing/done/failed |
| highlight_count | INT | 高光点总数 |

#### `highlights` — 高光点表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AI | 主键，AI分支续写时用它关联 |
| episode_id | INT FK | 所属剧集 |
| timestamp | INT | 触发时间点（秒） |
| scene_desc | TEXT | 场景描述（AI 生成，续写提示词用） |
| action_desc | TEXT | 角色动作描述 |
| emotion | VARCHAR | 情绪氛围 |
| visual_effect | VARCHAR | 视觉特效类型 |
| particle_type | VARCHAR | 粒子特效类型 |
| audio_cue | VARCHAR | 音效提示 |
| show_branch | TINYINT(1) | 是否弹出分支弹窗 |
| branch_options | TEXT(JSON) | 分支选项：`[{id:A, text:...}]` |
| interaction_type | TINYINT | 互动类型：1=手电 2=心跳 3=AI弹幕 NULL=未分析 |
| ai_danmaku | JSON | AI 弹幕列表，仅 interaction_type=3 时有值 |

#### `branch_results` — 分支续写缓存表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AI | 主键 |
| highlight_id | BIGINT FK | 所属高光点 |
| episode_id | INT | 所属集数（冗余，加速查询） |
| branch_id | VARCHAR(10) | 分支编号（A/B/C） |
| branch_text | VARCHAR(255) | 分支按钮显示文字 |
| ai_response | TEXT | AI 生成的续写文字内容 |
| result_type | VARCHAR(20) | 结果类型：`text`/`image`/`video` |
| media_path | VARCHAR(500) | 多媒体文件路径（图片/视频） |
| prompt_sent | TEXT | 发送给 AI 的完整 prompt（用于复盘） |
| token_usage | INT | AI 消耗 token 数 |
| created_at | DATETIME | 创建时间 |

**缓存设计说明**：`branch_results` 表同时承担"分支选项配置"和"AI 生成结果缓存"两个职责。相同 `highlight_id + branch_id` 组合只生成一次，后续请求直接返回缓存，避免重复调用 AI（降低成本 + 提高响应速度）。

## 6. API 接口清单

### 6.1 剧集接口

| 方法 | 路径 | 说明 | 请求体 | 响应 |
|------|------|------|--------|------|
| GET | `/` | 健康检查 | — | `{status, service, version}` |
| GET | `/episodes` | 获取所有剧集 | — | `EpisodeResponse[]` |
| GET | `/episodes/{episode_no}` | 获取单集详情 | — | `EpisodeResponse` |
| GET | `/episodes/{episode_no}/highlights` | 获取某集高光点（**App 核心接口**） | — | `HighlightResponse[]` |
| GET | `/episodes/{episode_no}/highlights/{id}` | 获取单个高光点 | — | `HighlightResponse` |

### 6.2 AI 分支接口

| 方法 | 路径 | 说明 | 方案 |
|------|------|------|------|
| POST | `/ai/branch` | AI 纯文字续写 | 方案A（保底） |
| POST | `/ai/branch-with-image` | AI 文字 + 配图 | 方案B（多模态） |
| POST | `/ai/branch-video` | 预渲染视频分支 | 方案C（演示用） |
| POST | `/ai/branch-options/{highlight_id}` | AI 生成分支选项 | 辅助 |

### 6.3 AI 互动类型分析接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/ai/analyze-interaction/{highlight_id}` | 分析单个高光点的互动类型 |
| POST | `/ai/analyze-interaction/batch/{episode_no}` | 批量分析某集所有高光点 |

## 8. 部署说明

### 8.1 本地开发环境

```bash
# 1. 安装依赖
cd D:\AI全栈\ep67-analysis\backend
pip install -r requirements.txt

# 2. 配置环境变量（复制模板，填入真实 Key）
cp .env.example .env
# 编辑 .env，填入 AI_API_KEY、MYSQL_PASSWORD 和 SERVER_BASE_URL

# 3. 启动 MySQL（确保 localhost:3306 可用，数据库 drama_pulse 已创建）

# 4. 启动服务（自动建表）
python main.py
# 服务运行在 http://localhost:8000
# API 文档：http://localhost:8000/docs
```

### 8.2 生产环境建议

- 使用 `gunicorn + uvicorn` 替代单进程 `uvicorn.run()`
- MySQL 使用独立服务器，不在本机
- `.env` 通过服务器环境变量注入，不放在代码仓库
- 多媒体资源（`multimodal_assets/`）建议迁移到对象存储（OSS/COS），减轻服务器压力
- `SERVER_BASE_URL` 改为公网域名，如 `https://yourdomain.com`
